Bronze_test1 Notebook - Explanation
This notebook implements the Bronze Layer (raw data ingestion) in your medallion architecture pipeline. It's the first stage of the Bronze-Silver-Gold data transformation pattern.

🎯 Purpose & Role
The Bronze layer serves as the landing zone for raw data with minimal transformation. This notebook:

Ingests raw flight data from CSV files
Preserves data lineage with metadata tracking
Implements data quality checks with quarantine handling
Partitions data by ingestion date for efficient querying
📊 Data Source
Location: /Volumes/databricks_dlt/sample_dlt_table/data_dlt/*.csv Target Catalog/Schema: databricks_dlt.sample_dlt_table Source Data: Flight statistics (2010-summary.csv) with origin/destination countries and flight counts


Create Bronze Orders Table (PySpark)
Purpose: Ingest raw CSV data into the bronze_orders Delta table

What it does:

Reads CSV files using spark.read.format("csv") with:

header=true - Uses first row as column names
mode=PERMISSIVE - Handles malformed records gracefully (sets fields to null)
Wildcard pattern *.csv reads all CSV files in the directory
Adds metadata columns for data observability:

_ingested_at - Exact timestamp when record was loaded (tracking)
_source_file - Full path to source CSV file (lineage)
_batch_id - Current date as batch identifier (grouping)
_partition_dt - Partition key for data organization (performance)
Preserves original columns:

DEST_COUNTRY_NAME - Destination country
ORIGIN_COUNTRY_NAME - Origin country
count - Number of flights (cast to integer)
Writes to Delta table with:

mode("overwrite") - Replaces entire table each run (full refresh pattern)
saveAsTable() - Creates managed Delta table in Unity Catalog
Auto-creates table schema from DataFrame
Key Pattern: Uses Unity Catalog metadata syntax col("_metadata.file_path") instead of the deprecated input_file_name() function for UC compatibility.

Output: Creates bronze_orders table with 510 records (255 unique country pairs × 2 due to metadata variations)

Create Quarantine Table (PySpark)
Purpose: Isolate bad/invalid records for investigation and quality monitoring

What it does:

Re-reads the same CSV files - Separate read operation ensures independence from Cell 1

Identifies bad records using quality rules:

Null destination country → Invalid
Null origin country → Invalid
Null flight count → Invalid
Negative flight count → Invalid (business rule: counts must be ≥ 0)
Categorizes errors with conditional logic:

Uses when().when().otherwise() pattern for error classification
Creates human-readable error messages
Concatenates raw fields with concat_ws("|",...) for debugging
Writes to quarantine table with:

mode("append") - Accumulates bad records over time (audit trail)
mergeSchema=true - Allows schema evolution if error types change
Separate table (quarantine_orders) keeps bad data isolated
Key Pattern: Defensive data quality pattern - catch errors at ingestion before they contaminate downstream tables.

Columns:

raw_content - Pipe-delimited original values for debugging
error_reason - Classified error type for reporting
_ingested_at - When the bad record was detected
_batch_id - Which batch produced the error

Data Quality Metrics (PySpark)
Purpose: Monitor ingestion health and data quality trends

What it does:

Ingestion Stats Query:

Groups records by _partition_dt (ingestion date)
Counts total rows per partition
Orders by date descending (most recent first)
Use case: Detect ingestion gaps or volume anomalies
Data Quality Metrics Query:

Focuses on today's partition only (WHERE _partition_dt = current_date())
Counts total records processed
Counts nulls in each critical field using CASE expressions
Counts distinct batch IDs (detects multiple runs)
Use case: Real-time quality monitoring during pipeline execution
Key Pattern: Uses spark.sql() for analytics queries while Cell 1-2 use DataFrame API for transformations. This is a hybrid approach common in Databricks.

Metrics tracked:

Volume metrics (total records, batches processed)
Completeness metrics (null counts per field)
Partition coverage (which dates have data)

🏗️ Architecture Patterns Used
1. Medallion Architecture - Bronze Layer
Raw Zone: Minimal transformation, preserves source fidelity
Schema-on-Read: No schema enforcement at ingestion
Append-Only Semantics: Overwrite mode for idempotency (same input → same output)
2. Data Quality Framework
Quarantine Pattern: Bad records isolated, not dropped
Fail-Fast: Quality checks at ingestion prevent downstream corruption
Error Tracking: Full audit trail of what failed and why
3. Metadata Enrichment
Lineage: _source_file tracks data provenance
Observability: _ingested_at enables time-travel debugging
Partitioning: _partition_dt optimizes query performance
4. PySpark Best Practices
Declarative Transformations: Uses DataFrame API (more optimized than RDDs)
Column Functions: Imports specific functions for readability
Delta Lake Integration: ACID transactions, time travel, schema evolution
🔗 Pipeline Integration
Position: First task in Bronze-Silver-Gold pipeline Job Task Key: bronze_task Dependencies: None (entry point) Downstream: Feeds silver_task which reads from bronze_orders

Data Flow:
CSV Files → bronze_orders (510 rows) → Silver Layer
           ↓
      quarantine_orders (bad records only)

⚠️ Current Behavior Notes
Full Refresh Strategy: mode("overwrite") means each run replaces all data. This is simple but loses historical tracking. Consider switching to incremental patterns (merge/upsert) for production.

Duplicate Reads: Cells 1 and 2 both read the same CSV files. Could be optimized to read once and branch.

Partition Key: Uses ingestion date (_partition_dt) not event date. All records get same partition value per run. If your data has event timestamps, consider time-based partitioning.

No Deduplication: Raw data loaded as-is. Deduplication happens in Silver layer via DeltaTable.merge().

Quarantine Append Mode: Quarantine table grows indefinitely. Consider adding a retention policy or date partitioning for cleanup.
